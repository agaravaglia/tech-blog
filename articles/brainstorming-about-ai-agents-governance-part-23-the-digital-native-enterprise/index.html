Brainstorming about AI agents governance (Part 2/3): The digital-native enterprise

How the agents governance framework adapts in digital-native companies with modern infrastructure, CI/CD culture, and self-service tooling.

Alessandro Garavaglia | 6 min read | Oct 8, 2025

In Part 1, I introduced a governance framework for AI agents based on three key pillars (Governance, Execution, and Supervision), with responsibilities spanning Strategy, Operations, and Technology. If you haven't read Part 1, I recommend starting here to understand the foundational concepts.

In this post, I will explore how this framework plays out in a digital-native enterprise, where modern infrastructure, unified data platforms, and developer-first cultures are already in place. The tech stack may be modern, but that doesn't mean governance is solved.

The digital-native context: fast, code-first, and data-rich

Digital-native companies operate in environments built for speed and automation. They typically have:

  • Modern data infrastructure with metadata, lineage, and classification built-in;
  • Unified governance across data, ML models, and analytics;
  • Developer-centric workflows, built around a deployment platform based on infrastructure-as-code, APIs, microservices, and CI/CD;
  • High expectations for automation, with fast, flexible, self-service tooling.

This context can be a dream for many developers, but it brings its own challenges. Some up-front decisions can help steer the planning of AI agent governance:

  • Leverage existing tools and processes from the deployment platform, reusing components and best practices (like tracing and monitoring tools, container approach, automated CI/CD);
  • Most agents will be internal-first, deployed within the company's infrastructure. External agents are used only when necessary.

Let's look at how the framework adapts in this context.

Governance

Agent registration and approval-as-code

The agent manifest, in the form of a structured YAML or JSON file, is embedded into the codebase and CI/CD pipeline. As explained in Part 1, it should contain:

  • The agent name, version, and purpose;
  • Tools and APIs it can access;
  • Risk classification;
  • Resource limits and deployment environment.

Agent creation follows Git-based workflows:

  • Developers submit a merge request with the agent manifest;
  • Bots run checks on compliance, scope, and risk;
  • For higher-risk agents, reviewers from the AI Governance Board are looped in.

Risk assessment doesn't need to be manual. Instead:

  • Use policy-as-code tools (like OPA, Rego, or custom linters);
  • Automatically score risk based on the agent's capabilities and access levels;
  • Define thresholds for auto-approval or escalation.

This way, low-risk agents fly through, while high-risk ones get the scrutiny they need.

Integration with Metadata & Lineage Systems

Agent manifests can refer to existing assets in the enterprise:

  • Data sources in the catalog;
  • Models in the ML registry;
  • Datasets classified as sensitive or regulated.

Since the enterprise is already aligned on a modern data stack, there is no need for reinvention, just tight integration with what's already there.

Ethics and bias checks

Even in fast-moving digital-native companies, bias and ethical risks cannot be ignored. The AI Governance Board should be responsible for:

  • Enforcing bias evaluations for agents interacting with customer data or making impactful decisions;
  • Reviewing potential ethical consequences of autonomous decisions, especially around fairness and inclusivity;
  • Requiring documentation for risk mitigations in the manifest (prompt reviews, fallback mechanisms, …).

These checks can be built into CI/CD via linting rules or enforced during approval steps.

Composition of the AI Governance Board

In digital-native settings, the AI Board can be lightweight but cross-functional. Suggestions for composition include:

  • A technical lead or architect (e.g., platform or ML team);
  • A security and privacy representative;
  • A legal representative;
  • A product or business stakeholder;
  • Optional: a rotating "ethics" reviewer for high-risk agents.

This encourages governance without creating bureaucracy.

Execution

This is where digital-native companies have the biggest advantage. The infrastructure is already there: containers, CI/CD, secrets management, IAM. Agent execution just needs to plug in.

Self-service AI Agent platform-as-a-service (PaaS)

With few lightweight components modelled as microservices, the internal platform can manage the agent lifecycle:

  • Accepts manifests, validates them, and deploys agents automatically;
  • Injects secrets, config, and environment variables;
  • Routes access via IAM roles or scoped credentials;
  • Provides a UI for exploring existing agents and requesting new ones.

The UI can also be exposed to non-technical business users who are responsible for monitoring the logic implemented by the agent. Requesting a new agent via the UI should trigger a merge request (as described above) to maintain consistency.

Internal-first deployment

Since the application landscape is modern and based on containers or microservices, AI agents are deployed inside internal infrastructure by default, similar to other microservices:

  • As containers on machines (VMs, Kubernetes, etc.);
  • As serverless apps (AWS Lambda, Cloud Functions, etc.);
  • As APIs served from internal endpoints.

External agent platforms are used only when necessary, and they follow the same governance checks. In such cases, custom code can deploy agents to third-party platforms via APIs.

Pre-built connectors

Avoid reinventing the wheel. The platform provides:

  • Secure, pre-approved connectors for third-party tools (Salesforce, Snowflake, Slack, Databricks, …);
  • Proxy interfaces or Model Context Protocol servers (MCPs) to standardize how agents interact with internal systems;
  • Shared logging, secrets, and CI/CD infrastructure.

The platform treats these services like any other applications within the enterprise, encouraging reuse of standards and best practices.

Supervision

Just because the systems are fast doesn't mean you can skip supervision. On the contrary, the more autonomous your agents, the more observability you need.

FinOps & business guidelines for agents

Digital-native teams apply FinOps principles, as for the rest of their cloud infrastructure:

  • Set budgets per team or agent;
  • Track cost per call, per tool, per model;
  • Trigger alerts or throttling when usage spikes.

In addition to FinOps, business stakeholders should review agent usage and logic regularly, ensuring alignment with business goals and avoiding scope creep. In flatter hierarchies like digital natives enterprises, this supervision can be lighter, but still necessary.

Observability by default

Both the budget guidelines and business KPIs require real-time monitoring. As with deployment, it's best to reuse existing observability infrastructure:

  • Structured logging with context: agent ID, prompt, tool usage;
  • Prompt tracing and chain-of-thought logs;
  • Metrics: latency, retries, LLM token usage, cost;
  • Dashboards to monitor agent health in real time.

This ensures integration between agentic logs and the rest of the enterprise applications.

Audit and compliance

With manifests and logs in place, compliance doesn't mean slowing down:

  • Every agent action is timestamped and logged;
  • Agent decisions can be replayed and explained;
  • Internal auditors and business teams can trace back decisions easily.

This supports both technical transparency and regulatory accountability.

Best practices and patterns

Best Practices

  • Treat manifests like config: validate them with CI/CD and policy-as-code tools;
  • Auto-approve low-risk agents: use risk scoring to scale approvals;
  • Build and maintain tool connectors: share across teams to reduce duplication;
  • Use data lineage for access scoping: automate permissions based on datasets;
  • Monitor cost and behavior centrally: build dashboards, alerts, and usage reports.

Watchouts

  • Prompt deterioration: without structure, agents become fragile and hard to debug;
  • Agent sprawl: if everyone builds their own version of the same agent, it will be chaos. The agent catalog needs to be visible and explorable by everyone;
  • Platform fragmentation: even with a great stack, inconsistent usage patterns can derail governance.

Conclusion

Modern infrastructure simplifies a lot, but it doesn't remove the need for governance. In digital-native companies, the real challenge isn't building the platform, but it is embedding governance into fast-moving, developer-driven workflows.

Ethical risks, compliance, and business alignment must still be considered, even in a tech-forward environment.

In Part 3 of this series, I will explore what happens in the opposite situation: a legacy-heavy and highly regulated enterprise.
