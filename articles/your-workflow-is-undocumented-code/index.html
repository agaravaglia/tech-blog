Your Workflow Is Undocumented Code

What I found out when I automated my publishing pipeline with Claude Code.

Alessandro Garavaglia | 5 min read | Feb 22, 2026

For a while, my publishing process worked fine. I wrote articles and edited them locally, then manually copied them online, and manually synchronize their publications with the GitHub repository if the article was about a demo. It worked. But every time I did it, I was running a mental checklist from memory.

At some point I thought about automating the publication. The obvious path was to use a platform with an API, and script against it. It was a clear and straightforward path, but sounded too easy. I wanted to learn something: build my own static site, own every file, understand what happens between a Markdown file and a live URL. The constraint was self-imposed, and made things interesting.

Once the site was mine, so was the full path from article to live page. And once I had that path end to end, I could write it down. This article is about what happened when I did.

AI assistants are getting a new kind of command

Over the past year, AI coding assistants have started converging on a common pattern: slash commands that trigger multi-step workflows. You describe what a workflow should do, and the assistant executes it as a sequence of steps. I use Claude Code for this (it calls them "skills"), but the idea is showing up across tools.

This is different from asking an AI to help you with a task: when you ask for help, the AI responds to you, but when you define a skill, the AI runs a procedure, and the skill file becomes the specification rather than the conversation. That shift in framing is subtle, but it changes how you think about what you are handing off.

So what did I specify?

The checklist was already there, just not written down

Before the skill existed, here is what publishing an article looked like:

  • Open the Markdown file, check the front matter (title, date, tags, all required)
  • Convert the body to HTML
  • Drop the output into the right folder
  • Update the article index (a JSON file the homepage reads from)
  • Rebuild the homepage
  • Commit and push

I had that sequence memorized and I ran it the same way every time, but it existed only in my head.

Writing the skill did not save me from thinking, instead it forced me to think properly once.

The act of defining the skill forced me to answer questions I had never explicitly asked. What are the actual steps, in order? What can vary between articles? What must never be skipped? I had answers to all of these, but I had never written them down. The skill file was the first time I did.

What the pipeline looks like

Here is the full sequence, in rough pseudo-code:

skill: /blog publish <article-path>

1. PARSE
   Read the Markdown file
   Extract front matter: title, date, tags, description, type, difficulty
   Validate all required fields — raise an error if any are missing

2. RENDER
   Convert Markdown body to HTML
   Inject into the article template (header, footer, styles, meta tags)
   Write output to tech-blog/articles/<slug>.html

3. INDEX
   Read the current index.json
   Append the new article entry (slug, title, date, tags, description)
   Sort all entries by date, newest first
   Write the updated index.json

4. HOMEPAGE
   Re-render index.html from the updated index.json
   Rebuild article cards, tag filters, and the recent articles section

5. PUSH
   git add the changed files
   git commit -m "Publish: <title>"
   git push to agaravaglia/tech-blog
   GitHub Pages picks it up and goes live

Five steps that look obvious on paper, but in practice I was skipping the sort in step 3, formatting tags differently depending on my mood, and occasionally pushing without updating the index at all. The skill does not care about any of that — it runs the same sequence every time, regardless of how tired I am.

The moment you have to specify every step, you stop glossing over the ones you usually skip.

Step 1 is also worth noting. The front matter validation is strict: if a required field is missing, the skill stops and raises an error. That felt annoying at first but prevents me from publishing articles with missing descriptions (and consequently the homepage card would be rendered blank).

A skill provides a fundamental shift to English as programming language.

What I actually learned

The honest version of this story is not "I saved time" (though I did). It is that writing the skill made me realize my manual process had been inconsistent in ways I had not noticed.

Tags were formatted differently between early and late articles. Some slugs matched the filenames, some did not. The homepage sort was not always by date. None of these were catastrophic, but they were the kind of thing that accumulates quietly until you sit down to write a specification and suddenly have to make a decision about each one.

The automation did not just speed up the workflow — it standardized it, so that every article published after the skill existed is consistent with every other one, not because I became more careful, but because the procedure stopped depending on my attention.

Automation means deciding, once, how the process should work, while maintaining yourself in control of the process.

Closing thoughts

Looking back, the most valuable thing about building my own static site was not the control, it was the excuse it gave me to understand every step of the path from Markdown to live URL, in a way I never had to when someone else's platform was handling it.

Once you understand each step well enough to describe it, you can write it down, hand it to a skill, and trust that the process will run the same way whether you are tired, distracted, or working through a backlog of articles you meant to publish weeks ago.

Every repetitive checklist you run from memory is already code, you just have not written it down yet.

This post is not meant to provide definitive answers or prescriptive guidance. Its goal is to spark reflections and discussions on the topic.
